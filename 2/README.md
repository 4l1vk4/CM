# CLI-прототип с конфигурацией

## Этап 1: Минимальный прототип с конфигурацией

### Общее описание
Данный проект представляет собой минимальный CLI-прототип, который загружает настраиваемые пользователем параметры из файла конфигурации формата .json.  
Цель этапа — создать рабочую версию приложения, способную принимать настройки и выводить их в формате `ключ: значение`, с обработкой возможных ошибок.

Программа реализована как консольное приложение (CLI) и поддерживает:
- загрузку конфигурации из JSON-файла,
- проверку наличия и корректности всех параметров,
- обработку ошибок (не найден файл, некорректный JSON, отсутствует параметр, неверный тип),
- вывод всех параметров пользователя.

---

### Функции и настройки

#### Настраиваемые параметры
- `package_name` — имя анализируемого пакета;
- `repository_url` — URL или путь к локальному тестовому репозиторию;
- `test_repo_mode` — режим работы с тестовым репозиторием;
- `package_version` — версия пакета;
- `ascii_tree_mode` — режим вывода зависимостей в формате ASCII-дерева (`true`/`false`);
- `filter_substring` — подстрока для фильтрации пакетов.

#### Основные функции
1. **CLI-интерфейс** — работает через командную строку, принимает имя JSON-файла с конфигурацией как аргумент.
2. **Загрузка и проверка конфигурации** — проверка наличия всех ключей и типов значений.
3. **Вывод параметров** — печать в формате `ключ: значение`.
4. **Обработка ошибок** — выводит понятные сообщения когда:
   - отсутствует файл конфигурации,
   - некорректном JSON,
   - отсутствует(ют) обязательный(е) ключ,
   - неверный тип значения.

---

### Установка и запуск
```bash
python 1st.py
python 1st.py config.json
python 1st.py not_found.json
python 1st.py err_val.json
python 1st.py 2_err.json
```
# Эмулятор для анализа зависимостей npm-пакетов
## Этап 2: Сбор данных

### Общее описание
На втором этапе приложение расширяется функциональностью получения информации о зависимостях npm-пакетов.  
Цель — реализовать логику загрузки и анализа данных о **прямых зависимостях** заданного пользователем пакета из локального репозитория или по URL-адресу GitHub.  

Приложение по-прежнему является консольным (CLI) и сохраняет все принципы предыдущего этапа — чтение конфигурации из JSON и обработку ошибок.

---

### Функции и возможности

#### **Основные функции**
1. **Работа в двух режимах**:
   - `local` — чтение данных из локального `package.json`.
   - `remote` — загрузка `package.json` с GitHub по указанному URL.

2. **Учёт версии пакета**:
   - Версия указывается в параметре `package_version`.
   - Если версия пакета не указана в `package.json`, программа сообщает:  
     `[Информация] У пакета нет явной версии, используется единственная доступная версия.`

3. **Извлечение зависимостей**:
   - Чтение поля `dependencies` из `package.json`.
   - Отображение всех прямых зависимостей пакета.
   - Возможность фильтрации зависимостей по подстроке (`filter_substring`).

4. **Обработка ошибок**:
   - Проверка корректности пути или URL.
   - Проверка наличия всех обязательных параметров.
   - Сообщения об ошибках формируются в человекочитаемом виде.

---

### Конфигурационный файл

#### Пример 1 — **Remote-режим**
```json
"package_name": "react",
"repository_url": "https://registry.npmjs.org/react/18.2.0",
"test_repo_mode": "remote",
"package_version": "18.2.0",
"ascii_tree_mode": false,
"filter_substring": ""
```
#### Пример 2 — **Local-режим**
```json
"package_name": "local-test",
"repository_url": "./pack1.json",
"test_repo_mode": "local",
"package_version": "1.0.0",
"ascii_tree_mode": false,
"filter_substring": ""
```
### Команды для запуска
```bash
python 2st.py config_local1.json
python 2st.py config_remote1.json
```

## Этап 3: Основные операции с графом зависимостей

### Общее описание
На третьем этапе цель проекта — построение графа зависимостей пакета с учётом транзитивности и выполнение основных операций над ним.  
Программа использует алгоритм BFS без рекурсии, корректно обрабатывает циклы зависимостей и поддерживает режим тестирования с локальным файлом описания графа.

---

### Основные функции

#### 1. Построение графа зависимостей (BFS)
- Алгоритм BFS (Breadth-First Search) используется для обхода всех зависимостей начиная с заданного пакета.
- Поддерживается фильтрация пакетов по подстроке, если указана в конфиге.
- Циклические зависимости корректно обрабатываются и не приводят к зацикливанию.

#### 2. Режим тестирования
- В режиме `"graph"` можно использовать локальный JSON-файл с описанием графа зависимостей.
- Пакеты могут называться любыми строками (например, большими латинскими буквами: `A`, `B`, `C`).
- Демонстрация функциональности на различных тестовых случаях.

#### 3. Фильтрация зависимостей
- Параметр `filter_substring` позволяет исключить пакеты, содержащие указанную подстроку.

---

### Конфигурационный файл

**Пример graph_conf1.json**
```json
"package_name": "A",
"repository_url": "graph1.json",
"test_repo_mode": "graph",
"package_version": "1.0.0",
"ascii_tree_mode": false,
"filter_substring": ""
```
### Команды для запуска
```bash
python 3st.py graph_conf1.json
```
